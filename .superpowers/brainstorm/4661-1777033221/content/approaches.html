Three approaches for the example database

All use Markdown + YAML frontmatter, KaTeX for math, Pagefind for search, and generate permalinks automatically.

A

Hugo (recommended)

Battle-tested static site generator with first-class taxonomy support — fields and tags become filterable taxonomies automatically. Extremely fast builds. Strong math support via KaTeX. Large community of math/academic sites built on it. Content model in YAML frontmatter, permalinks via slug field.

Pros

  • Built-in taxonomies (perfect for field/tag filtering)
  • Blazing fast, scales to thousands of examples
  • Mature, stable, widely used in academia
  • Single binary — easy to install and deploy

Cons

  • Go templating syntax is quirky to learn
  • Less flexible for custom UI components
B

Astro + Content Collections

Modern JS framework with typed content collections — define a Zod schema for your YAML frontmatter and get validation errors if you forget a required field. Great component model for building rich UI. Excellent Pagefind integration. More code to write upfront, but very flexible.

Pros

  • Schema-validated frontmatter catches mistakes at build time
  • Modern, component-based — easy to build rich filtering UI
  • TypeScript-native, great DX
  • Easy to extend toward community features later

Cons

  • More boilerplate to set up
  • Node.js dependency
C

Eleventy (11ty)

Minimal, highly configurable static site generator. You define everything yourself — good if you want total control, but requires more upfront design work. Good math support, Pagefind works well with it.

Pros

  • Extremely flexible — no opinions imposed
  • Lightweight, fast
  • JavaScript ecosystem

Cons

  • No built-in content schema or taxonomy system
  • More work to build filtering, browse-by-tag, etc.
  • Smaller community than Hugo/Astro for math sites
